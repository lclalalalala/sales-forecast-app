系统分层架构与核心计算逻辑

本文面向技术人员，说明系统的四层结构（前端层 / API 层 / 数据处理层 / 销量预测层）及其运行调用关系， 并解释数据处理、销量预测与补货量计算的基本逻辑。系统采用离线批处理模式：所有繁重计算在离线层完成并落表， API 层只做只读查询，前端只负责展示。

React + TypeScript Python Flask Pandas / NumPy 离线预计算表 (CSV) 预测目标：Units Sold

1分层架构总览

系统由四层组成。核心设计原则：计算与查询分离——预测、K 估算、安全库存、补货量、库存状态、排名等全部在离线数据处理层预计算， API 请求期间不做任何复杂计算，从而保证响应快速、结果可复现。

前端层 Frontend
页面渲染、筛选状态与路由、图表展示、下单草稿。不实现任何业务公式，只消费后端结果。
React · TypeScript · Tailwind · shadcn/ui · Recharts
API 层 Backend
参数校验、查询预计算结果表、关联维度表、组装 { context, data } 响应、返回空数据与错误状态。不执行预测 / K 估算 / 补货计算。
Flask · Blueprint · Repository（只读）
数据处理层 Offline Pipeline
离线批处理：清洗校验 → 构建维度/事实表 → 估算 K → 生成预测 → 误差统计 → 计算补货与库存状态 → 销售汇总/排名 → 落表。
offline_calculation/pipeline.py · Pandas · NumPy · YAML 配置
销量预测层 Forecast
被数据处理层调用的独立预测模块：集成模型（Ensemble）以 walk-forward 方式为每个 (门店, 商品) 生成未来 7 天点预测与预测区间。
forecast/batch.py · forecast/calculator.py（EnsembleForecastCalculator）

运行调用关系

系统存在两条时间上解耦的路径：

  • 离线批处理路径（写）：
    原始 CSV → 数据处理层 pipeline.run() →（内部调用销量预测层生成预测）→ 计算补货 → 写入 data/processed_data/*.csv 预计算表。
  • 在线请求路径（读）：
    前端 api.ts → HTTP/JSON → API 层 Query → Repository 只读查询预计算表 → 返回 { context, data } → 前端渲染。
前端层 (React) │ GET /api/overview | /rankings | /replenishment | /products/:id ▼ API 层 (Flask, 只读) │ Query.execute() → Repository.get_*() ▼ 预计算结果表 (data/processed_data/*.csv) ▲ │ pipeline 落表（离线，非请求期） 数据处理层 (offline_calculation) │ build_forecast() 调用 ↓ ▼ 销量预测层 (forecast.batch → EnsembleForecastCalculator)
实现现状提示：仓库中 api/app.py 为早期单体版本（直接在请求中调用旧的 ForecastService）。 当前前端消费的是新版离线批处理契约（{context,data}、lead_time_k、in_transit、safety_stock 等）， 对应 api/queries/ + api/infrastructure/ 的只读查询层。本文按新版架构描述。

2数据处理层的基本逻辑

数据处理层是一条离线流水线（offline_calculation/steps/*），以 Date + Store ID + Product ID 为最小粒度， 按 (store_id, product_id) 严格隔离，不混用不同门店/商品的数据。各步骤顺序执行并把结果写入预计算表。

加载与校验 load_validate —— 读取原始 CSV，校验主键唯一、必需字段存在、日期可排序、数值非负；生成数据质量报告。
→ data_quality_report
构建维度/事实表 build_fact —— 标准化字段与库存口径，生成日事实表。库存口径：当前(售后)库存 = Inventory Level − Units Sold（Inventory Level 为当天销售前库存）。
→ dim_store, dim_product, fact_daily_inventory_sales
估算提前期 K estimate_lead_time —— 对每个 (门店,商品) 用 MSE 估计下单到入库天数（详见 §4）。
→ fact_lead_time
生成预测 build_forecast —— 调用销量预测层，walk-forward 生成未来 7 天点预测与预测区间（详见 §3）。
→ fact_forecast
预测误差统计 build_error_stats —— 对固定 horizon 的历史预测误差做滚动标准差 σ，供安全库存使用。
→ fact_forecast_error_stats
计算补货与库存状态 build_replenishment —— 整合在途库存、K 天预测、安全库存，算出建议补货量与库存状态（详见 §4）。
→ fact_daily_replenishment
销售汇总与排名 build_sales_metrics / build_product_summary —— 计算日均销量、时间范围累计销量与畅销/滞销排名。
→ fact_daily_sales_metrics, fact_product_sales_summary

关键口径定义

指标定义
当前(售后)库存opening_inventory − units_sold
在途库存最近 K−1 个自然日的 Units Ordered 之和（不含当日）；K≤1 时为 0
日均销量最近 90 天累计 Units Sold / 90（缺失日按 0 补齐）
覆盖天数(当前库存 + 在途库存) / 日均销量
预测目标统一为 Units Sold（实际销量）；Demand 字段不作为对外展示指标。历史与预测在前端使用不同颜色/图例区分，避免混淆。

3销量预测层的基本逻辑

预测层是一个无需训练、即插即用的集成模型（EnsembleForecastCalculator）， 对异常值鲁棒、可解释性强。数据处理层以 walk-forward 方式逐 (门店, 商品) 调用它，产出未来 7 天每日点预测及预测区间。

3.1 集成预测（单日）

预测第 i 天（明天为 i=0）销量，由四个可解释分量加权组合：

ŷ = (0.3·近期均值 + 0.4·星期均值) × 趋势因子 × 0.7  +  0.3·近期均值 × 季节因子
分量含义
近期均值最近 14 天销量均值（捕捉当前水平）
星期均值历史同星期几最近 8 条均值（捕捉周内周期）；样本不足 4 条退化为近期均值
趋势因子clip(最近7天均值 / 最近30天均值, 0.7, 1.3)
季节因子clip(目标月份历史均值 / 近期均值, 0.7, 1.3)

结果做非负截断 max(0, ŷ)。裁剪区间 [0.7, 1.3] 防止趋势/季节过度反应。历史不足 7 天时触发降级预测（保守常数值）。

3.2 Walk-forward 与预测区间

对每个预测起点 forecast_origin_date 逐日滚动预测未来 7 天，并回填实际值与预测误差以供评估。预测区间由滚动误差标准差得到：

[lower, upper] = forecast ± 1.96 × σ_error   (lower 截断为 ≥ 0)

σ_error 为固定 horizon=3 的有效预测误差在 30 天窗口内的样本标准差（ddof=1）；有效误差不足 2 个时取默认 σ = 1.0。

重要约束：补货计算只使用点预测 forecast_units_sold，不使用预测区间上界。若某商品预测状态非 ready，该商品单独标记为"无法生成建议"，不影响其他商品。

4订单补货量的计算逻辑

补货计算在 build_replenishment 中完成，整合提前期 K、在途库存、K 天预测销量与安全库存，得到唯一的建议补货量。

4.1 提前期 K 估算

K 是下单到入库的提前期（(门店,商品) 级别，候选范围 0～7 天）。用库存平衡方程对每个候选 K 计算 MSE：

predicted_inventory[t+1] = Inventory[t] − UnitsSold[t] + UnitsOrdered[t−K]
MSE(K) = mean( (Inventory[t+1] − predicted_inventory[t+1])² )
  • 选择 MSE 最小的 K；MSE 相同时选较大的 K。
  • 无法估算时回退 K = 2；估得 K = 0 时按 K = 1 执行（记为 effective K）。
  • 返回 K 来源：estimated 或 default。

4.2 安全库存

safety_stock = Z × σ × √K   (Z = 2.33，约 99% 服务水平)

σ 为该 (门店,商品) 的预测误差滚动标准差（来自 §3.2 的误差统计）；K 为有效提前期。

4.3 建议补货量

补货目标是覆盖未来 K 天预测销量并留出安全库存，扣除手头与在途：

forecast_during_lead_time = Σ 未来 7 天预测的前 K 天点预测
raw = forecast_during_lead_time + safety_stock − current_inventory − in_transit_inventory
suggested_replenishment = ceil( max(0, raw) )

建议补货量始终为非负整数（向上取整）。

4.4 库存状态判定

基于含在途库存的覆盖天数分级（coverage_days = (current + in_transit) / 日均销量）：

覆盖天数状态
< 2 天紧缺 critical
≥ 2 且 < 4 天偏低 low
≥ 4 且 ≤ 7 天正常 normal
> 7 天充足 sufficient

日均销量为 0 时覆盖天数记为 N/A，状态为 undetermined（避免显示无穷大）。

销售表现 ≠ 库存风险：滞销 是销售标签，紧缺 是库存标签。一个商品可能"滞销但库存充足"，也可能"畅销且库存紧缺"，两者用独立字段表示，不合并成单一结论。

4.5 一个示例

某商品：K=3，未来 3 天预测合计 168，安全库存 40，当前库存 120，在途 30：

raw = 168 + 40 − 120 − 30 = 58  →  建议补货量 = 58 件
