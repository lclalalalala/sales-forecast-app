# 架构文档：门店销售、库存与补货决策系统

## 1. 上下文层（Context）

| 项目 | 内容 |
|------|------|
| 业务目标 | 为店长提供门店销售分析、库存风险识别和补货建议，帮助其在不查看原始数据的情况下决定重点商品和补货数量。 |
| 用户画像 | 店长；通过网页查看指定门店、时间范围、商品类别下的销售、库存、预测和补货结果。 |
| 运行模式 | 采用离线批处理。所有数据预先计算，API 请求只查询结果表，不执行复杂预测或补货计算。 |
| 数据范围 | 以 `Date + Store ID + Product ID` 为基本粒度，预测目标为 `Units Sold`。 |
| 约束条件 | 单用户、离线数据、无需登录和权限管理；当前不要求高并发和实时响应。 |
| 非目标 | 不包含用户登录、权限管理、真实采购下单、补货审批、数据导出、消息预警、多门店对比和供应商管理。 |
| 未来扩展 | 在线版本可在营业结束后重新校验数据、估算 `K`、生成预测和补货建议，并支持人工确认 `K`。 |

```text
                    │ 只读查询

                    │ 离线批处理写入

                    │
┌───────────────────┴────────────────────────┐
│              原始数据                         │
│ Kaggle CSV                                   │
│ 包含销售、库存、订货、商品和外部字段            │

```

## 3. 数据架构层（Data Architecture）

### 3.1 数据流图

```text
原始 CSV
  ↓
字段映射与类型标准化
  ↓
数据质量校验
  ↓
生成标准化日事实表
  ↓
估算并固定每个商品的 K
  ↓
Walk-forward 生成预测结果
  ↓
生成预测误差统计
  ↓
计算每日库存、覆盖天数和补货建议
  ↓
生成销售汇总和排名结果
  ↓
写入预计算查询表
  ↓
API 只读查询
  ↓
前端展示
```

## 4. 技术决策层（Tech Decisions）

| 决策 | 选择 | 理由 | 风险 |
|------|------|------|------|
| 前端框架 | React + TypeScript | 适合构建多页面分析界面，类型约束较好 | 需要维护前端状态 |
| 后端框架 | Flask | Demo 规模小，接口简单，Python 便于复用数据处理代码 | 大规模并发能力有限 |
| 数据处理 | Python + Pandas + NumPy | 适合 CSV 清洗、统计和批处理 | 大数据量时性能有限 |
| 查询存储 | csv | 无需数据库集群，读取预计算结果简单 | 不适合高并发写入 |
| 配置 | YAML | 业务参数集中、易于人工 Review | 配置修改需要重新批处理 |
| 预测计算 | 离线 | 结果可复现，API 不承担复杂计算 | 新数据不会自动即时更新 |
| 部署 | 本地 | 降低运维成本 | 暂无高可用能力 |

## 5. 质量约束层（Quality Gates）

### 5.1 批处理质量门禁

```yaml
data_pipeline:
  required_columns:
    - Date
    - Store ID
    - Product ID
    - Inventory Level
    - Units Sold
    - Units Ordered

  fail_on:
    - duplicate_primary_key
    - invalid_date
    - missing_required_column
    - negative_units_sold
    - negative_units_ordered
```

批处理必须在以下条件满足后生成查询表：

- 主键无重复；
- 必需字段存在；
- 日期可以排序；
- 数值字段类型正确；
- 库存关系校验通过；
- 生成的查询表记录数大于 0。

### 5.2 业务计算测试

至少测试：

- `closing_inventory = opening_inventory - units_sold`；
- `K` 候选范围为 `0～7`；
- MSE 相同时选择较大的 `K`；
- 无法估算时使用 `K = 2`；
- 补货量不能为负数；
- 补货量向上取整；
- 日均销量为 0 时库存状态为 `undetermined`；
- 销量相同时按 `Product ID` 稳定排序。

### 5.3 API 测试

每个主要接口至少覆盖：

- 正常查询；
- 不存在的门店；
- 不存在的商品；
- 无结果筛选；
- 非法时间范围；
- 缺少必需参数；
- 查询表为空；
- 数据版本不存在。

### 5.4 API 性能约束

由于 API 只查询预计算表：

```yaml
api:
  response_time_target_ms: 500
  complex_calculation_in_request: false
  readonly: true
```

### 5.5 前端约束

前端必须支持：

- 加载状态；
- 错误状态；
- 空数据状态；
- 数据截至日期展示；
- 预测销量与实际销量视觉区分；
- 当前门店和筛选条件展示；
- 从排名进入详情时保留上下文。
